# Contributing

Thanks for considering a contribution. A few things specific to this project,
because it does unsafe, architecture-specific, self-modifying-code tricks
where a "looks right" change can be subtly, silently wrong.

## Setup

```sh
git clone https://github.com/himanlabs/badgerpatch
cd badgerpatch
go build ./...
go test -v ./...
```

## Before opening a PR

- [ ] `go test ./...` passes locally
- [ ] `go vet -unsafeptr=false ./...` is clean (plain `go vet ./...` will flag
      the intentional uintptr->unsafe.Pointer conversion in `memory.go` —
      that's expected, see the doc comment there; CI uses the same flag)
- [ ] If you touched `jmp_amd64.go`, `jmp_arm64.go`, `copy_unix.go`,
      `copy_windows.go`, or `copy_darwin_arm64.go`: see the section below.
- [ ] New public functions have doc comments.
- [ ] If behavior changed, README examples still match.

## Changes to architecture-specific files (`jmp_*.go`, `copy_*.go`)

These files are the highest-risk part of the codebase: a bug here doesn't
fail to compile or fail a type check, it produces machine code that either
does nothing or corrupts execution — sometimes only on an architecture the
author didn't personally test on. Two of the two real bugs found in this
library's history lived here (an inlining-related silent no-op, and an
arm64 jump that dereferenced the wrong address). Please:

1. **Disassemble the generated bytes and paste the output in the PR.**
   Don't just eyeball the byte-shifting math. Example workflow used to
   verify the arm64 jump builder:

   ```sh
   # write a small program that calls buildJmpDirective(someAddr)
   # and dumps the resulting []byte to a file
   go run ./tools/gen_jump > /tmp/jump.bin
   aarch64-linux-gnu-objdump -D -b binary -m aarch64 /tmp/jump.bin
   ```

   Confirm the disassembly does what you intend — e.g. for a "jump to
   address X" builder, confirm the final instruction is a direct branch to
   the register holding X, not a load from the memory X points to.

2. **Test on the real architecture if at all possible**, not just via cross-
   compilation. CI covers linux/amd64 and linux/arm64 (via QEMU); darwin/arm64
   currently has no CI runner, so changes to `copy_darwin_arm64.go` need a
   manual test report from an Apple Silicon machine in the PR description.

3. **Add a regression test** that would have caught the bug you're fixing,
   where practical. For logic bugs this is usually possible; for "the CPU
   executes garbage and crashes" bugs, a disassembly check checked into the
   test suite (comparing generated bytes against a known-good byte sequence)
   is the practical equivalent.

## Reporting a bug

Please include:
- Go version (`go version`)
- OS/arch
- Whether the target function had `//go:noinline` (see README) — this is
  the first thing maintainers will ask, since an inlined target is the most
  common cause of "the patch didn't seem to apply."
- A minimal repro if at all possible.

## Code of conduct

Be respectful, assume good faith, keep discussion focused on the code.
